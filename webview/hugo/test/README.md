# webview/hugo/test

Test suite for the workstation-probe Hugo module. Each example site in
`../exampleSite/` imports the module from the local working tree (via a
relative `replace` directive in the site's `go.mod`). The runner builds
every site and diffs its output against the golden snapshot in
`expected/`.

## Run

```bash
# Verify mode — build each site, diff against expected/<site>/.
./run.sh

# Update mode — overwrite expected/<site>/ with the current build output.
# Use after intentional changes to the module's rendered output.
./run.sh --update
```

Exit code is 0 when every site matches, 1 when at least one diffs.

## Per-machine setup

The test site resolves the module import through Go's standard `replace`
directive in `../exampleSite/basic/go.mod`:

```go
replace github.com/assaneko/workstation-probe/webview/hugo => ../..
```

The path is **relative** (`../..` from the example site lands on the
Hugo module's directory), so cloning the repo to any home directory
needs no edits. There is no absolute path anywhere in version control.

## What is covered

`../exampleSite/basic/` exercises the shortcode's page-resource config
resolution path on the home page, plus a dev/demo workflow that talks
to a live probe server on the local machine.

| Page | Shortcode | Resolution path |
|---|---|---|
| `content/_index.md` | `{{< workstation-probe-panel "./localhost.yaml" >}}` | colocated home-page resource |

The single config uses `api: "http://localhost:19090"` and enables
`cpu`, `memory`, `gpu`, and `storage`, so the same file drives both the
snapshot test (no live data needed) and a live demo against the local
probe server (see below). When the probe server runs on a different
port, run `./scripts/dev-server.sh PORT=NNNN` instead and edit
`content/localhost.yaml` to match.

The snapshot in `expected/basic/` is regenerated by running
`./run.sh --update` whenever the rendered output changes
intentionally. Normal `./run.sh` runs are the contract — any
unexpected diff in the rendered HTML fails the test. Try a
`refresh: "5s" → "30s"` edit on a config to see the runner report a
clear diff pointing at the change.

## Live demo

The test sites are configured to talk to a probe server on
`localhost:19090` (the default port of `./scripts/dev-server.sh`).
Run them side by side to see the panels render real data:

```bash
# Terminal 1: probe server (foreground; Ctrl+C to stop)
./scripts/dev-server.sh

# Terminal 2: serve the basic test site
cd webview/hugo/exampleSite/basic
hugo server -D        # default: http://localhost:1313
```

Open <http://localhost:1313/>. The home page resolves the colocated
`./localhost.yaml`; the frontend does an HTTP fetch to
`http://localhost:19090/metrics`, parses the response, and renders the
local CPU, memory, GPU, and storage panels. CORS is enabled on the dev
probe server, so the cross-origin fetch from `localhost:1313` to
`localhost:19090` works.

`scripts/dev-server.sh` defaults to `GPU=auto`: it rebuilds the local
development server binary before starting, uses the `nvml` build tag when
`libnvidia-ml.so` is available, and falls back to the stub GPU collector
otherwise. Force a mode with `GPU=nvml ./scripts/dev-server.sh` or
`GPU=stub ./scripts/dev-server.sh`.

## Layout

- `../exampleSite/<name>/` — a self-contained Hugo site that imports the module
- `expected/<name>/` — golden snapshot of `../exampleSite/<name>/public/`
- `run.sh` — build + diff runner
- `.gitignore` — keeps build outputs out of the repo

Test sites are committed; their `public/` and `resources/` outputs are
gitignored. The `expected/` snapshots ARE committed — they are the
truth that tests verify against.
